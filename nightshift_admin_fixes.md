## 一、紧急修复（P0）

### 1.1 修改角色报错：`SyntaxError: Invalid or unexpected token (at admin:1:22)`

**现象**：管理员后台 → 用户管理 → 点击"修改角色"按钮，控制台报错 `Uncaught SyntaxError: Invalid or unexpected token (at admin:1:22)`，弹窗不弹出。

**根因**：`admin.html` 内联 JavaScript 存在语法错误，可能是最近一次修改（PIN 安全修复或角色权限修复）时引入的非法字符、括号不匹配、或字符串引号未闭合。

**修复要求**：
- 检查 `admin.html` 第 1 行附近（或内联 `<script>` 标签内）的 JavaScript 语法
- 重点检查修改角色相关的弹窗函数（如 `showRoleModal`、`changeRole` 等）
- 确保所有括号、引号、反引号成对闭合
- 修复后验证：点击"修改角色"能正常弹出角色选择弹窗，选择后调用 API 成功

**验收标准**：
- 控制台无 `SyntaxError`
- 点击"修改角色"弹出选择框（管理员/查寝员）
- 选择后调用 `/api/admin/user-update` 或类似接口，返回 200
- 页面刷新后角色已变更

### 1.2 操作日志始终显示"无日志记录"

**现象**：管理员后台 → 操作日志 → 无论选什么时间范围、什么操作类型，都显示"无日志记录"。

**根因排查方向**：
1. **前端查询参数错误**：`admin.html` 调用日志查询接口时，可能传了错误的时间范围或过滤条件
2. **后端接口返回空**：CloudBase `system_logs` 集合里确实没有数据（迁移后未产生新日志，或查询条件不匹配）
3. **前端渲染逻辑错误**：数据回来了但渲染函数有 bug，导致没显示

**修复要求**：
- 先确认 `system_logs` 集合里是否有数据（CloudBase 控制台直接查看）
- 如果集合有数据但前端不显示，检查 `admin.html` 里日志查询的 API 调用和渲染逻辑
- 如果集合无数据，确认是查询条件问题（如时间戳格式、字段名不匹配）还是确实没写入日志
- **注意**：之前 D1 迁移时 `system_logs` 的数据可能已导入，但新系统产生的日志写入的是 CloudBase。如果前端查询的是"最近90天"而数据库里只有迁移时的旧数据（时间戳可能不对），就会查不到

**验收标准**：
- 查询"全部操作"+"全部时间"能显示日志记录（至少显示迁移时的旧日志或新产生的日志）
- 日志列表显示：操作人、操作类型、时间、详情
- 切换时间范围（最近7天/30天/90天）过滤正常

### 1.3 系统设置页"最后更新时间"显示 `--`

**现象**：管理员后台 → 系统设置 → 敏感词库状态卡片中，"最后更新时间"显示 `--`。

**根因**：前端未正确读取 `settings` 集合中的 `last_update_time` 字段，或该字段不存在/格式不对。

**修复要求**：
- 检查 `admin.html` 中读取 `settings` 的代码，确认 `last_update_time` 字段的读取路径
- 如果 `settings` 集合中没有 `last_update_time`，需要在词库更新时（或初始化时）写入当前时间
- 时间格式建议：`YYYY-MM-DD HH:mm` 或相对时间（"2小时前"）

**验收标准**：
- 页面加载后"最后更新时间"显示具体日期时间（如 `2026-06-11 14:30`）
- 如果词库从未更新过，显示"从未更新"而非 `--`

## 二、功能增强（P1，汇报前尽量完成）

### 2.1 用户管理增加"删除用户"功能

**现状**：用户管理只有"修改角色"按钮，无法删除测试账号（如 `hacktest888`）或离职人员账号。

**需求**：
- 每个用户行增加"删除"按钮（红色，与"修改角色"并列或放在操作列）
- 点击删除后弹出确认框："确定删除用户 [用户名]？此操作不可恢复"
- 确认后调用后端 API 删除用户
- **后端权限校验**：仅 `admin` 角色可删除用户，不能删除自己（防止管理员自删导致系统无管理员）

**后端需要**：
- 新增或复用删除用户路由（如 `/api/admin/user-delete`）
- 校验：
  1. 当前用户 role === 'admin'
  2. 不能删除自己（`target_user_id !== current_user_id`）
  3. 可选：不能删除最后一个管理员（防止系统锁死）

**前端修改位置**：`admin.html` 用户列表渲染逻辑 + 新增删除按钮事件

**验收标准**：
- 管理员能看到删除按钮
- 点击删除有二次确认
- 删除后用户从列表消失
- 被删除用户无法再次登录
- 非管理员看不到删除按钮/调用接口返回 403

---

### 2.2 数据管理增加批量操作功能

**现状**：数据管理只能单个编辑/删除学生，无法批量修改状态（如大三全员设为离校）。

**需求**：
- 每行学生数据前增加复选框（`<input type="checkbox">`）
- 表头增加"全选"复选框
- 顶部或底部增加批量操作栏：
  - 批量修改状态：下拉选择（在校/离校/请假/未返校等）→ 点击"应用"
  - 批量删除：点击后二次确认
- 批量操作仅对勾选行生效

**后端需要**：
- 新增批量更新路由（如 `/api/admin/students-batch-update`）
- 接收参数：`ids: string[]`, `update: { status?: string, ... }`
- 使用 CloudBase `update` 批量更新（或循环单条更新，注意并发控制）
- 返回：成功数、失败数

**前端修改位置**：`admin.html` 数据管理 Tab 的表格渲染逻辑

**验收标准**：
- 能勾选多行（含全选/取消全选）
- 批量修改状态后，所有勾选行状态同步更新
- 批量删除后，勾选行从列表移除
- 操作后有 toast 提示（如"成功更新 15 人状态"）

## 三、技术约束

1. **只修改必要文件**：`admin.html`、`cloudbase/api/index.js`（如需新后端路由）
2. **不要改数据库结构**：复用现有 `users`、`system_logs`、`settings`、`dorm_students` 集合
3. **权限校验**：所有管理操作必须服务端校验 `role === 'admin'`，不能仅靠前端隐藏按钮
4. **保持现有 UI 风格**：新按钮/复选框使用现有 CSS 变量和 neumorphism 风格
5. **改完后更新 `sw.js` CACHE_NAME**（如 `v2026-0611f`），并执行 `npm run deploy` 部署前端
