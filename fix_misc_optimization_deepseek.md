# NIGHTSHIFT 查寝系统 — 杂项优化需求文档

**项目**：NIGHTSHIFT 查寝系统  
**目标**：8 项前端体验与交互优化  
**前提**：此前遗留的 4 项前端修复（报告报错、底部切换栏、卡片 UI、模式记忆）已处理完毕。

---

## 需求 1：筛选条件支持多选

**现状**：筛选条件（如年级、状态、宿舍等）目前为单选模式，只能同时激活一个条件。  
**期望**：支持同一维度内的多选（如同时筛选"大一"和"大二"，或同时筛选"在寝"和"请假"）。多选后结果取并集或交集（按业务逻辑，通常状态取并集、年级取并集）。  
**排查方向**：
- 检查当前筛选组件是原生 `<select>`、自定义下拉、还是按钮组。
- 若用按钮组，需将单选状态管理改为多选状态管理（数组/Set 存储激活项）。
- 筛选后的数据过滤逻辑需从 `===` 匹配改为 `.includes()` 或 `.some()` 匹配。
- UI 上需有"已选中"视觉反馈，并支持"清空筛选"或"全选"快捷操作。

## 需求 2：事假事由"工作室"支持二级展开

**现状**：选择人员状态为"事假"时，事由选择里有一项"工作室"，但无法进一步选择具体是哪个工作室。  
**期望**：点击/选择"工作室"后，展开二级选项（如：数据实战工作室、女工组、宣传组等，具体列表以实际业务为准）。用户选择具体工作室后，前端记录该明细用于内部逻辑，但**生成报告时统一显示"工作室"三个字，不暴露具体工作室名称**。  
**排查方向**：
- 事由选择组件需支持级联或二级展开交互。
- 数据结构上，事假状态可关联一个 `subReason` 字段存储具体工作室，但报告生成模块读取时只取一级事由"工作室"。
- 注意移动端展开后的触控区域不要与下方按钮重叠。

## 需求 3：模糊匹配逻辑全局采用

**现状**：模糊匹配（拼音首字母/模糊音/包含匹配）可能仅在部分输入框生效，其他输入场景无高亮提示。  
**期望**：**所有涉及人员姓名输入的文本框**（无论数据来源是手动输入、搜索框、还是任何其他输入方式）统一启用模糊匹配，并在下拉或联想列表中高亮匹配字符。  
**排查方向**：
- 梳理所有涉及姓名输入的交互点（顶部搜索、手动录入、筛选等）。
- 将现有的模糊匹配工具函数（拼音转码、模糊音映射）提取为全局公用模块。
- 确保匹配结果列表的渲染逻辑统一，高亮样式一致。

## 需求 4："外出"状态颜色替换

**现状**：当前"外出"状态的绿色用户反馈"太丑"。  
**期望**：将"外出"状态对应的绿色替换为 **#3c7a17** 或 **#15F5B9**，二选一以实际视觉协调为准（#3c7a17 偏深绿，#15F5B9 偏青绿）。  
**排查方向**：
- 全局搜索当前"外出"状态使用的颜色值（可能是硬编码色值或 CSS 变量）。
- 若使用 CSS 变量，修改变量定义；若硬编码，统一替换。
- 检查是否有 hover、active、边框、背景、文字等多处使用了该颜色，需一并替换保持统一。

## 需求 5：人员状态按账号持久化存储

**现状**：刷新页面后人员状态丢失，或单人查寝和多人查寝的状态互相干扰。  
**期望**：
- 人员标记状态（在寝/未归/请假/事假等）按**账号维度**持久化。
- 同一账号下，**单人查寝**和**多人查寝**分别独立保存状态（即两种模式互不覆盖）。
- 只要不点击"新查寝"按钮，即使刷新页面、关闭重开，状态也不应清空。
- 点击"新查寝"时，清空当前模式下的所有状态。  
**排查方向**：
- 存储介质可选 `localStorage`（键名建议包含账号标识和模式标识，如 `nightshift_state_${userId}_single`）。
- 页面加载时，先读存储恢复状态，再渲染 UI。
- 任何状态变更（点击在寝/未归/请假等）实时写入存储。
- "新查寝"按钮触发时删除对应存储键。
- 注意退出登录时清理该账号相关存储，避免隐私泄露。

## 需求 6：文案修改"包含实习学生"→"包含大三学生"

**现状**：界面上某处文案为"包含实习学生"。  
**期望**：改为 **"包含大三学生"**。  
**排查方向**：
- 全局搜索字符串"包含实习学生"，替换为"包含大三学生"。
- 检查是否有对应的变量名、注释、或后端返回的文案也需要同步（若后端返回，需确认后端是否已改或需要前端做映射）。

## 需求 7：登录界面紧凑化（移动端）

**现状**：登录页面在移动端占据整个视口高度，上下留白极少，视觉压抑，输入体验局促（见截图）。  
**期望**：整体布局更紧凑，减少不必要的垂直间距，让登录框在手机上看起来更舒服，不要顶满全屏。  
**排查方向**：
- 检查登录卡片的外边距/内边距（`margin`、`padding`），适当缩减。
- Logo 区域和标题区域可适当缩小尺寸或间距。
- 输入框高度和间距可微调。
- 确保修改后不影响桌面端布局（或用媒体查询仅作用于移动端）。
- 底部"注册新账户"区域不要被系统导航栏遮挡。

---

## 需求 8：多人查寝界面复用单人界面 + 轮询优化

**现状**：多人查寝单独写了一套界面，既难用又缺少事假等状态选项。  
**期望**：
- **多人查寝直接复用单人查寝的界面**，不要单独重写一套。在单人界面基础上增加"多人"特有的逻辑（如批量操作、消息汇总等）。
- 消息轮询间隔改为 **3 秒**。
- 人员状态轮询间隔改为 **30 秒**。
- 点击"刷新"按钮时**立即发送请求**，不要等待下一轮轮询。  
**排查方向**：
- 检查当前单人查寝和多人查寝是否已组件化，若未组件化，考虑将核心查寝逻辑（人员列表、状态按钮、报告生成）抽离为可复用组件。
- 多人查寝的差异点主要是：数据源为多个宿舍、消息聚合显示。这些可作为 props/参数传入。
- 轮询逻辑中，`setInterval` 的间隔值需调整；刷新按钮需独立调用一次请求，并可选地重置 interval 计时器。
- 确保轮询清理逻辑完善（页面卸载、切换模式时清除旧 timer，防止内存泄漏和重复请求）。
