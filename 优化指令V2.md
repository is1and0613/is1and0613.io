# NIGHTSHIFT 查寝系统 · 大版本优化指令 V2

> **执行原则**：本文档只描述需求、问题与排查方向，不指定具体文件内函数增删或行号。AI 执行者应根据现有代码结构自行实现。  
> **数据库**：Cloudflare D1（SQLite）。  
> **临时账户**：用户名 `chaqin`，密码 `123456`，初始化时写入数据库。  
> **聊天模式**：基于 D1 + HTTP 的"准实时留言板"（非 WebSocket 即时通讯），进入房间 / 手动刷新 / 低频轮询时同步消息与红点。

## Phase 1：Service Worker 优化

### 需求描述
当前 SW 已修复为 Stale-While-Revalidate，但仍存在以下隐患：
1. 旧缓存可能长期滞留，导致用户看到旧版页面或旧静态资源。
2. 用户无法感知系统已更新，旧 SW 长期不退出。
3. SW 拦截导航请求（HTML），曾导致多次 ERR_FAILED。

### 优化方向

**A. 版本化缓存与自动清理**
- SW 内定义版本常量（如 `CACHE_NAME = 'nightshift-v2026-0517-a'`）。
- **每次 deploy 前必须手动更新此版本号**（作为 deploy  checklist 第一项）。
- `activate` 事件触发时，遍历所有缓存，删除名称不等于当前版本号的旧缓存。

**B. 弱化导航拦截**
- `fetch` 事件中，对 `mode === 'navigate'` 的请求（即加载 HTML 页面）直接 `return` 不拦截，让浏览器始终从网络获取最新 HTML。
- 仅对 CSS/JS/图片/字体等静态资源做 Stale-While-Revalidate 缓存。
- 排查：确认拦截逻辑中没有对非 GET 请求（POST/PUT）返回缓存，否则会导致 API 请求被吞。

**C. 更新检测与提示**
- 注册 SW 后，监听 `updatefound` 与 `statechange`。
- 当新 SW 下载完成并进入 `installed` 状态，且当前已有旧 SW 在运行时，通过 Toast 提示用户"系统已更新，建议刷新页面"。
- 提示应持久显示（不自动消失），直到用户刷新。

**D. 移动端兼容**
- 排查：SW 的 `fetch` 拦截中，对失败的网络请求（如运营商网络到 Cloudflare 边缘节点超时）是否有 `.catch()` 兜底，避免返回 undefined 导致白屏。

## Phase 2：账户系统（D1 数据库）

### 需求描述
彻底废弃 `sessionStorage` 存储 Token 的临时方案，建立基于 D1 的正式账户体系。支持注册、登录、自动过期清理。不再区分"查寝端"与"组长端"，权限通过房间角色控制（创建者为房主，其他为成员）。

### D1 表结构规范（新建）

**表 1：users（用户表）**
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PRIMARY KEY | 自增 |
| username | TEXT UNIQUE NOT NULL | 用户名 |
| password_hash | TEXT NOT NULL | 密码哈希（bcrypt 或 Web Crypto PBKDF2） |
| display_name | TEXT | 显示昵称，默认同 username |
| created_at | DATETIME DEFAULT CURRENT_TIMESTAMP | 注册时间 |
| last_login_at | DATETIME | 最后登录时间 |
| is_temp | INTEGER DEFAULT 0 | 1=临时账户（chaqin） |

**约束**：
- 注册时 `username` 唯一性校验，已存在则返回错误。
- 密码禁止明文存储，必须哈希。
- 初始化时插入临时账户：`chaqin` / `123456`，`is_temp = 1`。

**表 2：rooms（查寝房间表）**
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PRIMARY KEY | 自增 |
| code | TEXT UNIQUE NOT NULL | 6 位数字/字母校验码 |
| creator_id | INTEGER NOT NULL | 创建者 user_id |
| created_at | DATETIME DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| expires_at | DATETIME NOT NULL | 过期时间（创建时 +90 分钟） |
| status | TEXT DEFAULT 'active' | active / expired / closed |
| dorm_building | TEXT | 关联宿舍楼（可选） |

**约束**：
- `code` 生成逻辑：随机 6 位（数字+大写字母，排除 0/O/I/1 等易混淆字符），生成后检查数据库唯一性，冲突则重试。
- `expires_at` 使用 SQLite `datetime('now', '+90 minutes')` 计算。
- 房间过期后 `status` 改为 `expired`，不可再加入，但历史数据保留供查看。

**表 3：room_members（房间成员表）**
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PRIMARY KEY | 自增 |
| room_id | INTEGER NOT NULL | 关联 rooms.id |
| user_id | INTEGER NOT NULL | 关联 users.id |
| joined_at | DATETIME DEFAULT CURRENT_TIMESTAMP | 加入时间 |
| role | TEXT DEFAULT 'member' | creator / member |
| last_read_msg_id | INTEGER DEFAULT 0 | 最后阅读的消息 ID，用于红点提示 |

**约束**：
- 同一用户同一房间不可重复加入（UNIQUE(room_id, user_id)）。

**表 4：room_states（房间人员状态表）**
| 字段 | 类型 | 说明 |
|------|------|
| id | INTEGER PRIMARY KEY | 自增 |
| room_id | INTEGER NOT NULL | 关联 rooms.id |
| student_name | TEXT NOT NULL | 学生姓名 |
| dorm_number | TEXT NOT NULL | 宿舍号（如 407） |
| bed_number | TEXT | 床位号（如 1） |
| status | TEXT DEFAULT 'present' | present / absent / leave / late |
| updated_by | INTEGER | 操作者 user_id |
| updated_at | DATETIME DEFAULT CURRENT_TIMESTAMP | 更新时间 |

**约束**：
- 联合索引 `(room_id, student_name)` 保证同一房间同一学生唯一一条记录。
- 初始化：创建房间时，从全局宿舍数据（现有 dorm-data API）拉取名单，批量插入此表，默认状态 `present`。

**表 5：room_logs（动作日志表）**
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PRIMARY KEY | 自增 |
| room_id | INTEGER NOT NULL | 关联 rooms.id |
| user_id | INTEGER NOT NULL | 操作者 |
| action_type | TEXT NOT NULL | 'status_change' / 'join' / 'leave_room' / 'system' |
| target_student | TEXT | 被操作学生姓名 |
| old_status | TEXT | 原状态 |
| new_status | TEXT | 新状态 |
| detail | TEXT | 额外描述（如事由） |
| created_at | DATETIME DEFAULT CURRENT_TIMESTAMP | 时间 |

**表 6：room_messages（房间消息表）**
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PRIMARY KEY | 自增 |
| room_id | INTEGER NOT NULL | 关联 rooms.id |
| user_id | INTEGER NOT NULL | 发送者 |
| content | TEXT NOT NULL | 消息内容 |
| created_at | DATETIME DEFAULT CURRENT_TIMESTAMP | 时间 |

### API 接口规范（新增/改造）

**POST /api/auth/register**
- Body: `{username, password, display_name?}`
- 校验：username 长度 3-20，password 长度 6-30，username 唯一。
- 返回：成功则返回 `{success: true, token}`，自动登录。

**POST /api/auth/login**
- Body: `{username, password}`
- 校验：先检查 `last_login_at`，若超过 30 天未登录，自动删除该用户并返回"账户已过期，请重新注册"。
- 更新 `last_login_at = CURRENT_TIMESTAMP`。
- 返回：Token。

**POST /api/auth/refresh**（可选）
- 延长登录态，更新 `last_login_at`。

**POST /api/room/create**
- Header: `Authorization: Bearer <token>`
- 生成 6 位校验码，插入 rooms 表，`expires_at` = now + 90 min。
- 将创建者插入 room_members，`role = 'creator'`。
- 从 dorm-data 拉取当前宿舍楼名单，初始化 room_states（全部 present）。
- 返回：`{success: true, code, expires_at}`。

**POST /api/room/join**
- Header: `Authorization: Bearer <token>`
- Body: `{code}`
- 校验：code 存在、status = active、未过期。过期返回"房间已过期"。
- 插入 room_members。
- 返回：`{success: true, room_id}`。

**GET /api/room/:code/sync**
- Header: `Authorization: Bearer <token>`
- 返回房间全量数据：
  - `states`: 所有学生当前状态列表
  - `logs`: 最近 50 条动作日志（按时间倒序）
  - `messages`: 最近 50 条消息（按时间正序，用于显示）
  - `members`: 当前在线成员列表（简化版：用户名+角色）
  - `room_info`: 创建者、创建时间、过期时间
- 此接口为"手动同步"核心，进入房间、点击"同步"按钮、低频后台轮询都调用它。

**POST /api/room/:code/state**
- Header: `Authorization: Bearer <token>`
- Body: `{student_name, new_status, detail?}`
- 更新 room_states 中对应记录。
- 插入 room_logs（action_type='status_change', old_status, new_status）。
- 返回：`{success: true}`。

**POST /api/room/:code/message**
- Header: `Authorization: Bearer <token>`
- Body: `{content}`
- 插入 room_messages。
- 返回：`{success: true, message_id}`。

**GET /api/room/:code/messages?last_id=<n>**（可选，用于增量拉取）
- 返回 `id > last_id` 的新消息，减少数据传输。

### 自动注销逻辑（惰性清理）
- 不依赖定时任务（Cloudflare Pages Functions 无原生 Cron）。
- 在 **login** 和 **refresh** 时检查：`last_login_at < datetime('now', '-30 days')`。
- 若过期：先删除该用户，返回 `success: false, message: '账户已过期，请重新注册'`。
- 临时账户 `chaqin` 不受此规则影响（`is_temp = 1` 时跳过清理）。

### 排查方向
- 现有 `auth.js` 从 JWT 校验改为 D1 查询，确认 `verifyToken` 在 `_utils.js` 中仍可用。
- 确认 D1 绑定：在 Cloudflare Dashboard → Pages → 项目 → D1 Database Bindings 中绑定数据库名（如 `DB`），代码中通过 `env.DB` 访问。
- 首次 deploy 前，在 D1 控制台执行初始化 SQL（建表 + 插入临时账户）。

## Phase 3：报告格式精简 + 实习开关

### 需求描述
原报告格式精简为 **3 种**：
1. **在校版**：正常在寝人员统计
2. **离校版**：请假/外出/已离校人员统计
3. **请假版**：详细请假事由与时间段统计

删除其他冗余格式。

### 新增：实习开关
- 在生成报告前，提供一个开关（Toggle / Checkbox）："是否包含实习学生"。
- 开启时：大三实习学生（标记为 `intern` 状态或特定标签）单独列出或纳入统计。
- 关闭时：实习学生从报告中排除，不计入应到/实到/未到人数。
- 实习学生数据来源： dorm-data API 中需增加 `is_intern` 或 `grade` 字段，或前端通过学号/年级规则判断（建议后端 dorm-data 增加字段，避免前端硬编码年级规则）。

### 排查方向
- 检查现有报告生成逻辑，确认被删除格式的代码已完全移除，无残留 DOM 或死代码。
- 实习开关状态应在生成报告时作为参数传入 API 或前端过滤函数。

## Phase 4：多人查寝模式

### 需求描述
登录成功后，不直接进入 index，而是进入**模式选择页**（可作为 index 的覆盖层或独立页面）。

**模式 A：单人查寝**
- 与现有逻辑一致，本地操作，数据不共享，不生成房间。
- 适合一个人快速查寝。

**模式 B：多人查寝**
- 进入多人查寝大厅，两个按钮：
  - **创建房间**：点击后生成 6 位校验码，显示"房间号 XXXXXX，有效期至 XX:XX"，并自动进入房间。
  - **加入房间**：输入 6 位校验码，验证通过后进入房间。

### 房间内交互设计

**布局**：
- 顶部：房间号、过期倒计时、当前成员头像/昵称列表。
- 中部：宿舍名单（列表式或卡片式，见 Phase 5）。
- 底部：操作栏（标记状态、生成报告、退出房间）。
- 右下角：悬浮消息按钮（显示红点未读数）。

**状态同步机制（非实时）**：
- 进入房间时：自动调用 `/api/room/:code/sync`，拉取全量状态、日志、消息。
- 手动同步：顶部或底部提供"刷新"按钮，点击重新拉取。
- 低频静默同步：可选每 30 秒自动调用一次 sync（频率远低于 3 秒，避免请求压力和卡顿），仅更新状态与消息。
- 状态更新：用户点击"在寝/未归/请假"后，**先乐观更新本地 UI**（立刻变色），同时 POST 到 API，失败则回滚并 Toast 提示。

**动作日志（实时显示在房间内）**：
- 在房间界面提供一个可展开/折叠的日志面板（或底部抽屉）。
- 每条日志格式："`[HH:MM]` `用户名` 将 `宿舍-床位 姓名` 状态由 `旧` 更新为 `新`"
- 日志数据来自 sync 接口的 `logs` 字段，按时间倒序排列。

**交流对话框（准实时留言板）**：
- 点击右下角悬浮按钮，弹出对话框（底部抽屉或侧边栏）。
- 显示历史消息（用户名 + 内容 + 时间）。
- 输入框发送消息 → POST → 成功后本地追加到消息列表（不等待 sync）。
- 红点提示：比较 `last_read_msg_id` 与最新消息 ID，有未读则按钮显示红点+数字。
- **明确限制**：这不是即时通讯，消息到达对方设备需要对方手动刷新或等待 30 秒轮询。适合查寝时的简短沟通（"407 灯坏了看不清"、"我去三楼了你们查四楼"）。

**冲突处理**：
- 两人同时修改同一学生状态：以最后一次写入为准（HTTP 模型限制）。
- 可在 sync 返回的数据中检查 `updated_at`，若本地操作后收到的 sync 数据中该学生 `updated_at` 更新，则提示"该学生状态已被他人更新，请确认"。

### 排查方向
- 确认单人模式与多人模式的数据流完全隔离：单人模式不走 room_states，继续使用本地逻辑或全局 dorm-data。
- 房间过期后（90 分钟或 status=expired），所有成员应被踢回模式选择页，并提示"房间已过期"。
- 退出房间（关闭浏览器/点击退出）时，是否保留房间数据？建议保留（数据在 D1），但成员标记为离线（通过 last_sync_at 判断，可选优化）。

## Phase 5：展示方式升级（列表式 + 卡片式 + 滑动切换）

### 需求描述
index / 房间内支持两种视图切换：

**列表式（默认）**：
- 现有样式优化，每行显示：宿舍号、姓名、床位、状态标记按钮。
- 适合快速批量操作。

**卡片式（新增）**：
- 以宿舍为单位展示大卡片，一屏一宿舍。
- 卡片内显示该宿舍所有成员头像/姓名/床位/状态。
- 支持 **Touch 左右滑动** 切换宿舍。

### 滑动切换实现方向
- 监听 `touchstart` 记录起始 X 坐标。
- `touchmove` 中实时计算位移，给当前卡片加 `transform: translateX()` 产生跟随手指的拖拽感。
- `touchend` 判断：
  - 位移 > 屏幕宽度 30% 且速度足够 → 切换下一宿舍，卡片滑出动画。
  - 位移 < -30% → 切换上一宿舍。
  - 否则 → 回弹到原位。
- 切换时更新顶部宿舍标题与页码指示器（如 "3/12" 或圆点）。
- 提供左右箭头按钮作为辅助（桌面端鼠标点击，移动端也可点）。

### 排查方向
- 滑动与垂直滚动（页面上下滚动）冲突：判断 `touchmove` 中水平位移与垂直位移的比值，若垂直 > 水平，则放弃滑动切换，交给页面滚动。
- 滑动时状态标记按钮的点击事件误触：确保滑动过程中（手指未抬起）不触发按钮 click，仅在 `touchend` 且无显著位移时放行点击。
- 卡片式下搜索/筛选功能是否可用？建议搜索时自动切回列表式，或卡片式仅支持宿舍级筛选。

## Phase 6：UI 全面去 Emoji + 冲突排查

### 需求描述
全站禁止使用 Emoji 作为图标或状态标识。所有 Emoji 替换为以下方案之一：
- **SVG 图标**：内嵌 `<svg>` 或引用外部图标库（如 Phosphor Icons、Heroicons，轻量且无需 npm）。
- **文字标签**：如"在寝""未归"直接显示文字，不用 🟢🔴。
- **CSS 图形**：如用 `::before` + `border-radius` 做状态圆点。

### 排查清单（重点检查区域）
- **Toast 图标**：成功/错误/信息三种 Toast 的图标。
- **状态标记**：在寝、未归、请假、迟到等状态指示器。
- **按钮图标**：上传、刷新、返回、搜索、设置等按钮。
- **空状态**：无数据时的占位图标（如 📭 → 用 SVG 空盒子）。
- **加载动画**：Spinner 用 CSS 动画，不用 ⏳ 或 🔄。
- **manual-upload 中的假单类型图标**：离校、事假、外出等标识。

### UI 冲突排查方向
- **z-index 层级**：检查弹窗（Modal）、Toast、Loading、抽屉（Drawer）、对话框的层级关系。要求：Loading < 弹窗 < Toast < 抽屉/对话框（最高）。
- **固定定位元素重叠**：底部操作栏（fixed bottom）与 iOS 底部安全区（env(safe-area-inset-bottom)）的兼容。
- **输入框被软键盘顶起**：移动端点击输入框（如加入房间输码、聊天输入）时，页面是否自动滚动避免遮挡。
- **长按选中文本**：确认已全局设置 `user-select: none` 或仅允许输入框选择，避免移动端长按出现系统菜单与自定义菜单冲突。
- **颜色对比度**：主色 `#2C436F` 与背景 `#EAF0E2` 的文本可读性，确保 WCAG 基础对比度。

## Phase 7：数据库初始化与部署 Checklist

### D1 初始化 SQL（首次执行）
按 Phase 2 的表结构建表，并执行：
```sql
INSERT INTO users (username, password_hash, display_name, is_temp, last_login_at) 
VALUES ('chaqin', '<hash_of_123456>', '临时账户', 1, CURRENT_TIMESTAMP);
```

### Deploy 前检查项
1. **SW 版本号**已更新。
2. **D1 Bindings** 已在 Cloudflare Dashboard 配置（变量名如 `DB`）。
3. **环境变量** `.dev.vars` 与线上 Dashboard 一致（JWT_SECRET 等）。
4. **node_modules 不在 git 中**（`.gitignore` 检查）。
5. **临时账户** 可登录。
6. **单人/多人模式切换** 流畅。
7. **去 Emoji** 全局搜索无残留（可搜索 `.` 后接 Unicode 范围或人工抽查）。

## 附录：技术限制诚实声明

| 功能 | 限制 | 实际体验 |
|------|------|----------|
| 房间消息 | 无 WebSocket，基于 HTTP 轮询 / 手动刷新 | 发送后，对方需手动刷新或等待 30s 轮询才能看到。非微信式即时通讯，适合查寝场景简短留言。 |
| 状态同步 | 同上 | 两人同时改同一学生，后写入覆盖前者，可能产生短暂不一致，sync 时提示冲突。 |
| 自动注销 | 无定时任务，采用惰性清理 | 用户超过 30 天未登录，**下次登录时**才发现账户被删。期间数据仍占库，但极小。 |
| 房间过期 | D1 级，非内存级 | 90 分钟后房间标记 expired，但已加入的成员若未刷新仍可看到旧页面，刷新后被踢出。 |
