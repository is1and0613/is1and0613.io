## 模块一：manualupload 文本解析与模糊匹配逻辑重构

### 1.1 区队名排除规则

**问题描述**
当前 manualupload 的文本解析采用全局关键词贪婪匹配，不区分词语在文本中的角色。关键词 "网安" 会被映射到事由 "网安工作室"，但输入文本中的 "网安2501"、"网安2401" 等是**专业区队标识**，不应被识别为事由。

**具体需求**
- **排除规则**：当匹配文本中出现 "网安" + 数字（如网安2501、网安2401、网安2301）时，**排除**该匹配，不映射到 "网安工作室" 事由。
- **保留规则**：纯 "网安"（无后续数字/字母）仍可作为 "网安工作室" 事由触发。
- **扩展性**：将规则设计为可配置数组，方便后续添加其他专业（如 "信安2401"、"情报2501" 等）的排除模式。

**排查方向**
- 定位 manualupload 中负责 "文本→姓名/事由" 解析的核心逻辑（通常位于 OCR 结果处理或假单文本分析模块）。
- 检查当前是否已有任何正则匹配逻辑，避免重复造轮子。

### 1.2 人名模糊匹配：有约束的相似度算法

**问题描述**
OCR 或人工输入难免出现形近字错误，如 "孔怡罪" 应为 "孔怡霏"、"黄品睿" 应为 "黄思睿"，当前精确匹配无法兼容此类误差。但若仅按姓氏匹配，会导致大量误匹配（如所有姓 "张" 的都高亮），反而增加核对负担。

**具体需求**
- **算法建议**：采用编辑距离（Levenshtein Distance）或汉字拼音相似度，但需叠加以下约束：
  - **姓氏锁定**：首字必须相同（严格匹配），杜绝跨姓氏误配。
  - **长度约束**：目标姓名与输入姓名长度差异不得超过 1 个字。
  - **阈值控制**：
    - 2 字姓名：最多允许 1 个字符差异。
    - 3 字姓名：最多允许 1 个字符差异（可考虑放宽到 2 个，但需提高相似度权重）。
  - **优先级**：精确匹配始终优先于模糊匹配；模糊匹配结果需标记为 "待确认"。
- **视觉高亮**：在 manualupload 的识别结果列表中，模糊匹配命中的人员姓名需使用**醒目但不过度打扰**的样式区分：
  - 建议：橙色/琥珀色边框（`#A99563` 迪奥金）或浅金色背景，文字色保持高对比。
  - 避免：红色（与 "未归/异常" 状态混淆）、纯荧光色（深色模式下刺眼）。
- **交互提示**：鼠标悬停（或移动端长按）时显示 Tooltip："模糊匹配：输入 '孔怡罪' → 系统姓名 '孔怡霏'，请人工确认"。

**排查方向**
- 确认姓名数据源（Supabase/D1 本地缓存）的读取方式，确保模糊匹配在本地完成以减少请求延迟。
- 验证高亮样式在深色模式下的可读性。

### 1.3 分组标题保护（Bad Case：数分 / 孔怡霏 曾子蕙）

**问题描述**
manualupload 的文本解析当前不区分"分组标题"与"事由描述"。当用户输入格式为"分组标题 + 换行 + 人名"时，分组标题（如"数分"、"网安"、"高数"、"信安"等）会被误识别为请假事由。

**Bad Case 复现**
- **输入文本**：
  ```
  数分
  孔怡霏 曾子蕙
  ```
- **当前结果**：事由 = "数分"，分组名 = "数分 (2/2人)"
- **预期结果**：事由 = 未识别/留空（由查寝人员后续手动选择），分组名 = "数分 (2/2人)"

**具体需求**
- **分组标题优先规则**：如果输入文本的第一行（或单独一行的短文本，2-4字）后紧跟着人名列表（同一行或下一行），则该行优先视为分组标题/区队标识，**不参与事由匹配**。
- **事由上下文感知**：事由匹配不应仅基于"词是否存在"，而应基于上下文：
  - "因数分课请假" → 提取"数分"为事由（有明确事由语境）
  - "数分\n孔怡霏" → "数分"为分组标题（后接人名，无语境）
- **排除列表扩展**：除了带数字的区队名，纯文本的区队/专业简称（如"数分"、"信安"、"情报"等）若出现在人名列表附近，也应被排除从事由匹配。
- **歧义处理**：若无法判断语境，优先视为分组标题（人名匹配是核心功能，事由可后续人工补选）。

**排查方向**
- 检查分组名提取逻辑：当前分组名"数分"是从哪里来的？是用户输入的第一行，还是事由匹配回写的？
- 检查"文本→分组名/事由/人名"的三段式解析顺序。

### 1.4 事由层级联动错误（Bad Case：数分一级事由 误高亮"合唱团"具体事由）

**问题描述**
用户输入文本中，"数分"被正确识别为一级事由（事假），但展开的具体事由列表中，"合唱团"被错误地高亮选中。"数分"和"合唱团"属于完全不同的一级事由分支，不应产生联动。

**Bad Case 复现**
- **输入文本**：`数分 / 孔怡霏 曾子蕙`
- **当前结果**：
  - 一级事由："数分" ✅（正确高亮）
  - 具体事由："合唱团" ❌（被错误高亮，与"数分"毫无关联）
  - 底部自定义输入框出现"数分"（疑似"其他"具体事由的文本回写）
- **预期结果**：
  - 一级事由："数分" ✅
  - 具体事由：无默认选中 / 或根据"数分"正确映射到其子事由（如"学习"、"补课"等，若配置中存在）
  - "合唱团"不应被高亮，它属于独立的一级事由分支（如"社团活动"或"文艺"）
  - 底部"其他"自定义输入框仅在用户主动选择"其他"具体事由后出现，且默认留空

**根因排查方向**
1. **数据层**：检查 D1/Supabase 中事由层级表，确认"数分"的子事由列表里是否被错误地写入了"合唱团"。
2. **索引残留**：若具体事由采用数组索引选中，可能一级事由切换时，具体事由的选中索引未重置，导致上一组用户的"合唱团"索引残留。
3. **默认回显逻辑**：检查是否存在"默认选中第一个具体事由"或"默认选中上次选择"的逻辑，且该逻辑未与当前一级事由做联动校验。
4. **回写污染**：检查底部自定义输入框的"数分"是用户主动输入还是系统错误回写。若是系统回写，需修复回写逻辑，避免一级事由名称被重复填充到"其他"具体事由的文本框中。

**具体需求**
- **联动校验**：切换一级事由时，具体事由的选中状态必须完全重置，不允许跨一级事由的选中状态残留。
- **数据隔离**：确保每个一级事由的具体子事由列表严格独立，"数分"下不应出现"合唱团"、"篮球队"等无关项。
- **回显溯源**：若底部自定义输入框出现"数分"，需确认来源。若是系统回写，需修复回写逻辑，避免一级事由名称被重复填充到"其他"具体事由的文本框中。
- **数据清理**：检查事由层级数据表，清理循环引用和重复项，确保一级事由与具体事由无交集。

**验收标准**
- [ ] 选择一级事由"数分"后，具体事由列表仅显示"数分"的真实子项，"合唱团"不出现且不被高亮。
- [ ] 切换一级事由（如从"数分"切到"篮球队"），原具体事由选中态完全清除，新列表无残留高亮。
- [ ] 底部"其他"自定义输入框仅在用户主动选择"其他"具体事由后出现，且默认留空，不回写一级事由名称。

## 模块二：新拟态 UI 全局改造（浅色 + 深色两套）

### 2.1 整体方案

保留主题切换功能，将浅色和深色两套主题**全部**改造为新拟态风格。浅色用参考图那种灰白浮雕感，深色用暗色凹凸阴影。

**浅色新拟态配色**
- 主背景：`#E0E5EC`
- 卡片背景：`#E0E5EC`（与背景同色，靠阴影区分层次）
- 凸起阴影亮部：`#FFFFFF`
- 凸起阴影暗部：`#A3B1C6`
- 嵌入阴影方向相反
- 文字主色：`#4A5568`
- 文字次色：`#718096`
- 强调色：保留迪奥金 `#A99563` 或调整为 `#8B7D6B`（适配浅色）

**深色新拟态配色**
- 主背景：`#1C312C`（松林绿）
- 卡片背景：`#1C312C`（与背景同色，靠阴影区分层次）
- 凸起阴影亮部：`rgba(255,255,255,0.08)`
- 凸起阴影暗部：`rgba(0,0,0,0.5)`
- 嵌入阴影方向相反
- 文字主色：`#E8E7E2`
- 文字次色：`#8A9A8C`
- 强调色：`#A99563`（迪奥金）

### 2.2 阴影参数规范（浅色）

**凸起元素（卡片、按钮）**
```
box-shadow:
  8px 8px 16px #A3B1C6,
  -8px -8px 16px #FFFFFF;
```

**嵌入元素（输入框、搜索框）**
```
box-shadow:
  inset 6px 6px 12px #A3B1C6,
  inset -6px -6px 12px #FFFFFF;
```

**小元素（图标按钮、圆形按钮）**
```
box-shadow:
  4px 4px 8px #A3B1C6,
  -4px -4px 8px #FFFFFF;
```

**按下态（active）**
```
box-shadow:
  inset 3px 3px 6px #A3B1C6,
  inset -3px -3px 6px #FFFFFF;
```

### 2.3 阴影参数规范（深色）

**凸起元素（卡片、按钮）**
```
box-shadow:
  6px 6px 12px rgba(0, 0, 0, 0.5),
  -6px -6px 12px rgba(255, 255, 255, 0.08);
```

**嵌入元素（输入框、搜索框）**
```
box-shadow:
  inset 4px 4px 8px rgba(0, 0, 0, 0.5),
  inset -4px -4px 8px rgba(255, 255, 255, 0.08);
```

**小元素（图标按钮、圆形按钮）**
```
box-shadow:
  3px 3px 6px rgba(0, 0, 0, 0.5),
  -3px -3px 6px rgba(255, 255, 255, 0.08);
```

**按下态（active）**
```
box-shadow:
  inset 2px 2px 4px rgba(0, 0, 0, 0.5),
  inset -2px -2px 4px rgba(255, 255, 255, 0.08);
```

### 2.4 组件映射（两套统一）

| 组件 | 新拟态处理方式 |
|------|---------------|
| 宿舍卡片（`.dorm-card`） | 凸起效果，圆角保持 `16px`，取消传统扁平阴影 |
| 底部操作栏按钮 | 凸起效果，按下时转为嵌入 |
| 搜索框/输入框 | 嵌入效果，聚焦时外框加微光（浅色：`0 0 0 2px rgba(169,149,99,0.3)`；深色：`0 0 0 2px rgba(169,149,99,0.5)`） |
| 楼层标签/筛选按钮 | 凸起效果，选中态（active）转为嵌入或叠加迪奥金边框 |
| 导航箭头按钮 | 圆形凸起，按下嵌入 |
| 宿舍卡片 header | 不做凸起，保持扁平但压暗（避免层次过多），深色下用 `#1A1D3C`，浅色下用 `#E0E5EC` 微亮 |
| 状态标签 | 见 2.5 特殊处理 |

### 2.5 状态标签特殊处理（关键）

**用户明确要求**：状态标签（在寝/离校/事假/外出/未归）不套新拟态阴影，保留高对比度扁平样式，但增加**彩色小点**作为状态提示。

**具体设计**
- **整体样式**：扁平化标签，细边框，圆角 `4px`，背景透明或极低透明度。
- **彩色小点**：标签左侧（或文字前）放置一个 `6px-8px` 的圆形色点，颜色与状态对应：
  - 在寝：迪奥金 `#A99563`
  - 离校：紫色 `#9B59B6`
  - 事假：橙色 `#F39C12`
  - 外出：青绿 `#15F5B9`
  - 未归：红色 `#E74C3C`
- **选中态**：标签背景填充对应颜色，文字反白（白色或深色），小点可隐藏或变为白色边框圆点。
- **为什么**：警务查寝场景需要快速扫视状态，新拟态的低对比度阴影会干扰信息读取。彩色小点提供即时色觉锚点，同时保持界面整洁。

**排查方向**
- 检查当前 `.status-tag` 的样式定义，确认是否有 `box-shadow` 需要移除。
- 确认状态标签颜色在深浅两套主题下是否统一。

## 模块三：深色模式全局化与对比度修复

### 3.1 全局深色覆盖

**问题描述**
深色模式目前仅在 index（查寝主界面）生效，login、upload、manualupload 仍为浅色，夜间使用刺眼。且 index.css 中存在大量硬编码颜色（`background: white`、`color: #1a1a2e` 等），导致 theme.css 的深色变量无法完全透传。

**具体需求**
- **范围**：login（登录页）、upload（假单上传页）、manualupload（手动输入/识别结果页）、index（查寝主界面）全部接入深色模式。
- **CSS 变量统一**：检查 `index.css`，将所有硬编码颜色替换为 `var(--xxx)`，确保深色变量能正确透传。重点检查：
  - `.search-section` 背景色
  - `.floor-tabs` 背景色
  - `.dorm-card`、`.dorm-header`、`.dorm-nav` 背景色
  - `.student-item` 边框色
  - `.status-bar` 背景色
  - `.filter-bar` 背景色
  - 搜索框 `input` 背景与文字色
  - 所有 `border-bottom: 1px solid #xxx` 硬编码
- **状态同步**：用户切换主题后，通过 localStorage 持久化，所有页面读取同一键值，确保跨页面体验一致。
- **旧配色清理**：index.css 中残留的旧主色 `#2C436F`、旧强调色 `#FDD95F` 需替换为当前配色变量。

**排查方向**
- 搜索各页面硬编码的颜色值（`#fff`、`#ffffff`、`white`、`#000` 等），替换为 CSS 变量。
- 确认 login/upload/manualupload 的 HTML 是否引入了与 index 相同的 CSS 变量/主题类名。

---

### 3.2 卡片 Header 亮度压制

**问题描述**
index 中宿舍卡片式布局的 header（顶部标题栏/宿舍号区域）在深色模式下亮度过高，与松林绿 `#1C312C` 背景割裂，视觉突兀。

**具体需求**
- 将卡片 header 背景统一为绀色 `#1A1D3C` 或更深的松林绿 `#152822`。
- header 文字（宿舍号）使用迪奥金 `#A99563` 或浅灰白 `#E8E7E2`，确保对比度。
- 若 header 使用了渐变或阴影，降低透明度或改用深色渐变。
- 浅色模式下 header 保持与卡片背景同色（`#E0E5EC`），靠微阴影区分层次。

**排查方向**
- 确认卡片 header 的具体选择器，查看其背景色/渐变定义。
- 检查是否有 `background: #fafafa` 或 `background: white` 写死在 header 样式中。

---

### 3.3 对比度全面巡检

**问题描述**
深色模式下部分文字、图标、按钮对比度不足，导致辨识困难。

**检查项**
- 所有页面输入框的 placeholder 文字。
- 按钮禁用态（disabled）的文字与背景。
- 状态标签（在寝/离校/事假/外出/未归）在深色模式下的边框与文字。
- 底部统计栏（"当前显示 4 人..."）的数字与标签。
- Toast 提示文字。
- 上传页/手动输入页的识别结果表格/列表文字。
- 搜索高亮文字（`.search-highlight`）。

**标准**
关键文字与背景对比度至少达到 WCAG AA 标准（4.5:1），大字号/粗体可降低到 3:1。

**排查方向**
- 使用浏览器 DevTools 的对比度检查工具扫描各页面。
- 重点检查 `theme.css` 中 `--text-muted: #6A7A6E` 在 `#1C312C` 背景上的对比度是否达标。

## 模块四：验收标准汇总

### 4.1 manualupload 验收
- [ ] 输入 `"网安2501 孔怡罪 黄品睿"`，系统**不**将 "网安2501" 识别为 "网安工作室" 事由。
- [ ] 将 "孔怡罪" 模糊匹配到 "孔怡霏" 并高亮提示（橙色/金色边框）。
- [ ] 将 "黄品睿" 模糊匹配到 "黄思睿" 并高亮提示。
- [ ] 输入 `"张溪影"`（已存在精确匹配），不触发模糊匹配高亮。
- [ ] 输入 `"张三"`（假设系统有 "张溪影"、"张笑怡"），因仅姓氏相同，**不**触发模糊匹配。
- [ ] 输入 `"数分\n孔怡霏 曾子蕙"`，"数分"仅作为分组标题，事由保持未识别/留空。
- [ ] 输入 `"因数分课请假\n孔怡霏"`，"数分"被识别为事由（语境明确）。
- [ ] 选择一级事由"数分"后，具体事由列表仅显示"数分"的真实子项，"合唱团"不出现且不被高亮。
- [ ] 切换一级事由（如从"数分"切到"篮球队"），原具体事由选中态完全清除，新列表无残留高亮。
- [ ] 底部"其他"自定义输入框仅在用户主动选择"其他"具体事由后出现，且默认留空，不回写一级事由名称。

### 4.2 深色模式验收
